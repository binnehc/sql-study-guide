Day 01 · Foundation

SQL BASICS

SQL (Structured Query Language) is how you talk to a database. Every query follows the same basic anatomy — you tell the database what to get, where to get it from, and how to filter/sort it.

SELECT

Which columns to return. Use * for all columns, or list specific ones.

FROM

Which table to read from. Always required.

WHERE

Filter rows before any grouping. Only rows matching the condition are kept.

ORDER BY

Sort the results. ASC (default) = smallest first. DESC = largest first.

LIMIT

Cap the number of rows returned. Useful for previewing data or "top N" queries.

DISTINCT

Remove duplicate rows from the result. Place right after SELECT.

Here's the table we'll use as our example throughout Day 1:

employee_idnamedepartmentsalaryhire_date
1AliceEngineering1200002020-03-15
2BobEngineering950002021-06-01
3CarolMarketing880002019-11-20
4DaveMarketing720002022-01-10
5EveEngineering1100002020-08-05
SQL · Basic Queries
-- Get all columns from all rows
SELECT * FROM employees;

-- Get only specific columns
SELECT name, salary FROM employees;

-- Filter: only Engineering employees
SELECT name, salary
FROM   employees
WHERE  department = 'Engineering';

-- Sort by salary, highest first
SELECT name, salary
FROM   employees
ORDER BY salary DESC;

-- Filter + Sort + Limit: top earner in Engineering
SELECT name, salary
FROM   employees
WHERE  department = 'Engineering'
ORDER BY salary DESC
LIMIT  1;
Essential WHERE operators
IN / NOT IN

Match against a list of values. WHERE dept IN ('Eng', 'Mktg') is cleaner than multiple OR conditions. NOT IN excludes a list.

BETWEEN

Inclusive range filter. WHERE salary BETWEEN 80000 AND 120000 is equivalent to >= 80000 AND <= 120000.

LIKE

Pattern matching on strings. % = any characters, _ = exactly one character. WHERE name LIKE 'A%' finds names starting with A.

AND / OR / NOT

Combine conditions. AND = both must be true. OR = either is true. Use parentheses to control precedence: (a OR b) AND c.

SQL · Filter Operators
-- IN: match multiple values (cleaner than OR)
SELECT * FROM employees
WHERE  department IN ('Engineering', 'Marketing');

-- NOT IN: exclude a list
SELECT * FROM employees
WHERE  department NOT IN ('Finance');

-- BETWEEN: inclusive range
SELECT * FROM employees
WHERE  salary BETWEEN 80000 AND 120000;

-- LIKE: % matches any string, _ matches one character
SELECT * FROM employees
WHERE  name LIKE 'A%';      -- names starting with A

-- AND + OR: combine conditions (parentheses control order)
SELECT * FROM employees
WHERE  (department = 'Engineering' OR department = 'Marketing')
  AND  salary > 90000;

Query Execution Order

SQL does NOT run in the order you write it. The real order is: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. This matters — it's why you can't use a SELECT alias inside a WHERE clause.

Practice

Write a query to return the name and salary of every employee in the Engineering department, sorted by salary from highest to lowest.

Table: employees (employee_id, name, department, salary, hire_date)

SQL
Answer
SELECT   name, salary
FROM     employees
WHERE    department = 'Engineering'
ORDER BY salary DESC;

INNER JOIN — only matching rows from both tables

The most common join. Returns a row only when the key exists in both tables. Dave (NULL dept) and Finance (no employees) are both excluded.

SQL
SELECT e.name, d.dept_name
FROM       employees   e
INNER JOIN departments d ON e.dept_id = d.dept_id;
namedept_name
AliceEngineering
BobEngineering
CarolMarketing

Dave (NULL dept) → excluded. Finance dept (no employees) → excluded.

LEFT JOIN — all rows from the left table, matched or not

Returns every row from the left table (employees). If there's no match in the right table, the right-side columns are NULL. Use this to find "things without a match" — e.g., employees with no department.

SQL
SELECT e.name, d.dept_name
FROM      employees   e
LEFT JOIN departments d ON e.dept_id = d.dept_id;

-- Find employees with NO department:
SELECT e.name
FROM      employees   e
LEFT JOIN departments d ON e.dept_id = d.dept_id
WHERE     d.dept_id IS NULL;   -- the "no match" trick

Mirror of LEFT JOIN. Returns every row from the right table (departments). Finance shows up even though no employees belong to it. Rarely used — most developers just swap the table order and use LEFT JOIN instead.

The union of LEFT and RIGHT. Returns every row from both tables. Non-matching sides get NULL. Use when you need a complete picture across two tables regardless of matches.

The "Missing Data" Pattern

LEFT JOIN + WHERE right.id IS NULL is one of the most common interview patterns. It finds rows in the left table with no match on the right — e.g., "customers who never placed an order", "users with no activity."

Practice

Write a query to find all employees who have no department assigned. Return only the employee name.

Tables: employees (id, name, dept_id) · departments (dept_id, dept_name)

SQL
Answer
SELECT    e.name
FROM      employees   e
LEFT JOIN departments d ON e.dept_id = d.dept_id
WHERE     d.dept_id IS NULL;

Day 03 · Summarising Data

GROUP BY & HAVING

Aggregation functions collapse many rows into a single summary value. GROUP BY controls which rows are grouped together before the summary is computed.

COUNT(*)

Number of rows in each group. COUNT(col) ignores NULLs.

SUM(col)

Total of a numeric column. NULLs are ignored.

AVG(col)

Mean value. NULLs are excluded from both numerator and denominator.

MIN / MAX

Smallest or largest value in the group.

SQL · Aggregations
-- Count employees per department
SELECT   department, COUNT(*) AS headcount
FROM     employees
GROUP BY department;

-- Average salary per department
SELECT   department, AVG(salary) AS avg_salary
FROM     employees
GROUP BY department;
WHERE vs HAVING — the most common interview gotcha
WHERE — filters BEFORE grouping
SQL
-- Filter individual rows first
SELECT   department, COUNT(*) AS cnt
FROM     employees
WHERE    salary > 80000    ← rows removed here
GROUP BY department;

Keeps only high-salary employees, then groups them. The group itself can have any size.

HAVING — filters AFTER grouping
SQL
-- Filter the group summary
SELECT   department, COUNT(*) AS cnt
FROM     employees
GROUP BY department
HAVING   COUNT(*) > 1;    ← groups removed here

Groups all employees first, then removes groups with only 1 person. You can only use aggregate functions here, not raw column values.

Rule of thumb

If your filter is on an individual row value → use WHERE.
If your filter is on the result of an aggregate (COUNT, SUM, AVG…) → use HAVING.

Two more aggregation gotchas
COUNT(DISTINCT col)

Counts only unique non-NULL values. Classic pattern: "how many unique users placed an order?"

SQL
-- Distinct departments vs total employees
SELECT
  COUNT(*)                     -- 5 rows
, COUNT(department)            -- 5 (no NULLs here)
, COUNT(DISTINCT department)  -- 2 unique depts
FROM employees;
SELECT column not in GROUP BY

Every column in SELECT must either be in GROUP BY or wrapped in an aggregate. This is the #1 GROUP BY syntax error.

SQL
-- ❌ ERROR: name not in GROUP BY
SELECT   department, name, COUNT(*)
FROM     employees
GROUP BY department;

-- ✓ Fix: either aggregate name or add to GROUP BY
SELECT   department, COUNT(*)
FROM     employees
GROUP BY department;
Practice

Write a query to show each department, its headcount, and average salary — but only for departments where the average salary is above 90,000.

Table: employees (employee_id, name, department, salary)

SQL
Answer
SELECT   department,
         COUNT(*) AS headcount,
         AVG(salary) AS avg_salary
FROM     employees
GROUP BY department
HAVING   AVG(salary) > 90000;

Day 04 · Edge Cases & Patterns

NULLS, SUBQUERIES
& SELF-JOINS

Part 1 · NULL Handling

NULL means "unknown" or "missing" — it is not zero, not an empty string. NULL has its own rules that trip up almost every SQL beginner.

NULL ≠ NULL

You cannot compare NULL with =. WHERE col = NULL always returns nothing. Always use IS NULL or IS NOT NULL.

COUNT(*) vs COUNT(col)

COUNT(*) counts all rows. COUNT(col) counts only non-NULL values in that column. This is a common interview trap.

NULLs in JOINs

NULL never matches anything — not even another NULL. A row with a NULL key will never match in a JOIN condition.

COALESCE()

COALESCE(col, 0) returns the first non-NULL value. Use it to substitute a default when a column is NULL.

SQL · NULL Gotchas
-- WRONG — this returns 0 rows, not the NULLs
SELECT * FROM employees WHERE dept_id = NULL;

-- CORRECT
SELECT * FROM employees WHERE dept_id IS NULL;

-- COUNT difference: employees table has 4 rows, 1 has NULL dept_id
SELECT COUNT(*)         -- returns 4  (all rows)
     , COUNT(dept_id)   -- returns 3  (ignores NULLs)
FROM employees;

-- Replace NULL with a default
SELECT name, COALESCE(dept_id, 0) AS dept_id
FROM   employees;
Part 2 · Subqueries

A subquery is a query nested inside another query. They work but can get hard to read — CTEs (Day 5) are the cleaner alternative. Knowing subqueries helps you understand legacy code and interview questions.

SQL · Subqueries
-- In WHERE: employees earning above the average
SELECT name, salary
FROM   employees
WHERE  salary > (SELECT AVG(salary) FROM employees);

-- In FROM: treat result as a temporary table (like a CTE)
SELECT dept, avg_sal
FROM (
    SELECT department AS dept, AVG(salary) AS avg_sal
    FROM   employees
    GROUP BY department
) AS dept_avgs
WHERE  avg_sal > 90000;
Part 3 · Self-Joins

A self-join joins a table to itself. This sounds strange but is very useful for comparing rows within the same table — classic examples: finding employees who earn more than their manager, or finding pairs of products bought together.

-- Alias the same table twice — "e" is the employee, "m" is their manager
SELECT
    e.name        AS employee,

Self-Join Pattern

Always give the table two different aliases (e.g., e and m). SQL then treats them as if they were two separate tables. The join condition defines the relationship between the two copies.

Practice

Write a query to find employees whose salary is higher than their manager's salary. Return the employee name, their salary, and their manager's salary.

Table: employees (employee_id, name, salary, manager_id) — manager_id references employee_id

SQL
Answer

    e.name   AS employee,
    e.salary AS emp_salary,
    m.salary AS mgr_salary
FROM       employees e
INNER JOIN employees m ON e.manager_id = m.employee_id
WHERE      e.salary > m.salary;

Day 05 · Readable Queries

COMMON TABLE
EXPRESSIONS

A CTE is a named temporary result set defined at the top of your query. It only exists for the duration of that query. The main benefit: breaking complex logic into readable, named steps instead of deeply nested subqueries.

Without CTE — hard to read
SQL
SELECT department
FROM (
  SELECT department,
         SUM(salary) AS total
  FROM   employees
  GROUP BY department
) dept_totals
WHERE total > 200000;
With CTE — clean and named
SQL
WITH dept_totals AS (
  SELECT department,
         SUM(salary) AS total
  FROM   employees
  GROUP BY department
)
SELECT department
FROM   dept_totals
WHERE  total > 200000;

CTEs are essential for window functions. You cannot filter on a window function result in the same query — you must wrap it in a CTE first, then filter in the outer query.

SQL · Why CTEs Enable Window Functions
-- This FAILS — you can't use ROW_NUMBER alias in WHERE of same query
SELECT *, ROW_NUMBER() OVER (...) AS rn
FROM   employees
WHERE  rn = 1;          -- ❌ ERROR: rn doesn't exist yet

-- This WORKS — CTE saves the result, outer query filters it
WITH ranked AS (
    SELECT *, ROW_NUMBER() OVER (
        PARTITION BY department
        ORDER BY     salary DESC
    ) AS rn
    FROM employees
)
SELECT name, department, salary
FROM   ranked
WHERE  rn = 1;          -- ✓ top earner per department

You can chain multiple CTEs with a comma — each can reference the ones above it:

SQL · Chained CTEs
WITH

step_one AS (
    SELECT ... FROM raw_table
),

step_two AS (
    SELECT ... FROM step_one   -- references the first CTE
    WHERE  ...
)

SELECT * FROM step_two;
Practice

Rewrite this using a CTE: find departments where total salary exceeds 200,000.

SQL
Answer

    SELECT   department,
             SUM(salary) AS total
    FROM     employees
    GROUP BY department
)
SELECT department
FROM   dept_totals
WHERE  total > 200000;

Day 06 · Advanced Analytics

WINDOW FUNCTIONS

Window functions calculate across a set of rows without collapsing them. Unlike GROUP BY which gives one row per group, window functions add a new column to every existing row.

Core Syntax

function() OVER ( PARTITION BY col ORDER BY col )
PARTITION BY = reset the calculation for each group  ·  ORDER BY = define the sequence within each partition.

ROW_NUMBER vs RANK vs DENSE_RANK

All three assign a number to each row within a group. The difference appears when there are ties (equal values).

ROW_NUMBER()

Always unique. Ties get arbitrary order. No gaps. Use when you need exactly one row per group.

RANK()

Ties get the same rank. The next rank skips. E.g., two 2nds → next is 4th. Creates gaps.

DENSE_RANK()

Ties get the same rank. The next rank does NOT skip. E.g., two 2nds → next is 3rd. No gaps.

Example — ranking employees by salary within each department:

namedeptsalaryROW_NUMBERRANKDENSE_RANK
AliceEng120000111
EveEng110000222
BobEng11000032 (tie)2 (tie)
CarolEng8000044 (gap!)3 (no gap)
DaveMktg900001 ← resets11
GraceMktg72000222
SQL · All Three Ranking Functions
SELECT
    name,
    department,
    salary,
    ROW_NUMBER()  OVER (PARTITION BY department ORDER BY salary DESC) AS row_num,
    RANK()        OVER (PARTITION BY department ORDER BY salary DESC) AS rnk,
    DENSE_RANK()  OVER (PARTITION BY department ORDER BY salary DESC) AS dense_rnk
FROM employees;
Other Useful Window Functions
SUM() OVER()

Running total or group total. Add ORDER BY to make it cumulative.

LAG() / LEAD()

Access a value from the previous or next row. Great for "change from last month" patterns.

NTILE(n)

Divide rows into n equal buckets. NTILE(4) creates quartiles.

FIRST_VALUE() / LAST_VALUE()

Get the first or last value in the window. Useful for comparing each row against the group leader.

SQL · Running Total + LAG Example
-- Running salary total per department (ordered by hire date)
SELECT
    name, department, salary,
    SUM(salary) OVER (
        PARTITION BY department
        ORDER BY     hire_date
    ) AS running_total
FROM employees;

-- Salary change vs previous employee hired in same dept
SELECT
    name, department, salary,
    LAG(salary) OVER (
        PARTITION BY department
        ORDER BY     hire_date
    ) AS prev_salary
FROM employees;
The "Top N per Group" Pattern — most common interview question

"Return the top-earning employee per department" — this pattern appears constantly. The trick: ROW_NUMBER in a CTE, then filter in the outer query.

SQL · Top 1 per Department
WITH ranked AS (
    SELECT
        name, department, salary,
        ROW_NUMBER() OVER (
            PARTITION BY department
            ORDER BY     salary DESC
        ) AS rn
    FROM employees
)
SELECT name, department, salary
FROM   ranked
WHERE  rn = 1;   -- change to rn <= 3 for "top 3 per group"

ROW_NUMBER vs DENSE_RANK here

If ties exist and you want both tied employees at rank 1, use DENSE_RANK() ... WHERE rn <= 1. If you want exactly one row even when tied, use ROW_NUMBER(). The choice depends on the problem statement.

CASE WHEN — SQL's if/else

CASE WHEN evaluates conditions and returns a value. It can appear in SELECT (to label/transform), inside aggregates (to count/sum conditionally), or in ORDER BY.

SQL · CASE WHEN Patterns
-- 1. Label/categorise rows
SELECT name, salary,
    CASE
        WHEN salary >= 100000 THEN 'Senior'
        WHEN salary >= 75000  THEN 'Mid'
        ELSE                        'Junior'
    END AS level
FROM employees;

-- 2. Conditional count (count only rows that match)
SELECT
    COUNT(CASE WHEN salary >= 100000 THEN 1 END) AS senior_count,
    COUNT(CASE WHEN salary  < 100000 THEN 1 END) AS junior_count
FROM employees;
Practice

Write a query to rank employees by salary within each department (highest = rank 1) using DENSE_RANK. Show name, department, salary, and rank.

Table: employees (employee_id, name, department, salary) — use DENSE_RANK so ties don't create gaps

SQL
Answer

    name,
    department,
    salary,
    DENSE_RANK() OVER (
        PARTITION BY department
        ORDER BY     salary DESC
    ) AS salary_rank

Whenever you see "number rows within a group, then aggregate/filter based on that number", use this template:

Template

Easy Filter Google Ads

Return the ad_id and name of all ads that are active, have more than 500,000 impressions, and were last updated in 2024.

ads
columntype
ad_idinteger
namevarchar
statusvarchar — 'active', 'paused', 'deleted'
impressionsinteger
last_updateddate
Approach
  1. SELECT ad_id, name — only the columns asked for.
  2. Filter status = 'active' with WHERE.
  3. Add AND impressions > 500000.
  4. Add AND EXTRACT(YEAR FROM last_updated) = 2024 for the year filter.
SQL
Solution
SELECT  ad_id, name, impressions
FROM    ads
WHERE   status = 'active'
  AND   impressions > 500000
  AND   EXTRACT(YEAR FROM last_updated) = 2024;

Why EXTRACT?

You could also write last_updated >= '2024-01-01' AND last_updated < '2025-01-01' — this is actually faster on indexed columns because it avoids wrapping the column in a function. Both are correct for an interview.

Easy Most Popular Search Category

For each month in 2024, count the total searches per category. Return category_name, month, and total_searches, ordered by month then by count descending.

searches
columntype
search_idinteger
user_idinteger
search_datedate
category_idinteger
categories
columntype
category_idinteger
category_namevarchar
Approach
  1. JOIN searches to categories on category_id to get the category name.
  2. Filter to 2024: WHERE EXTRACT(YEAR FROM search_date) = 2024.
  3. Use DATE_TRUNC('month', search_date) to group by month.
  4. GROUP BY category_name and the truncated month, then COUNT(*).
  5. ORDER BY month ASC, total_searches DESC.
SQL
Solution
SELECT   c.category_name,
         DATE_TRUNC('month', s.search_date) AS month,
         COUNT(*) AS total_searches
FROM     searches   s
JOIN     categories c ON s.category_id = c.category_id
WHERE    EXTRACT(YEAR FROM s.search_date) = 2024
GROUP BY c.category_name,
         DATE_TRUNC('month', s.search_date)
ORDER BY month ASC, total_searches DESC;
Medium Google Maps Flagged UGC

Find the single place category with the highest number of "off-topic" user-generated content reviews. Return the category name and its off-topic count.

maps_ugc_review
columntype
review_idinteger
place_idinteger
content_tagvarchar — e.g. 'off-topic', 'relevant'
place_info
columntype
place_idinteger
place_categoryvarchar — e.g. 'restaurant', 'park'
Approach
  1. JOIN maps_ugc_review to place_info on place_id.
  2. Filter: WHERE content_tag = 'off-topic'.
  3. GROUP BY place_category and COUNT(*).
  4. ORDER BY count DESC then LIMIT 1 to get the top category.
SQL
Solution
SELECT   p.place_category,
         COUNT(*) AS off_topic_count
FROM     maps_ugc_review r
JOIN     place_info      p ON r.place_id = p.place_id
WHERE    r.content_tag = 'off-topic'
GROUP BY p.place_category
ORDER BY off_topic_count DESC
LIMIT    1;

LIMIT 1 vs subquery

LIMIT 1 is the simplest approach. An alternative is wrapping in a subquery with WHERE off_topic_count = MAX(off_topic_count), but LIMIT 1 is cleaner and equally valid in an interview.

Medium Ad Campaign Performance

Calculate the average cost-per-click (CPC) for each campaign and ad group combination. CPC = total cost ÷ total clicks. Round to 2 decimal places.

ad_clicks
columntype
click_idinteger
datedate
campaign_idinteger
ad_group_idinteger
clicksinteger
costdecimal
Approach
  1. GROUP BY campaign_id, ad_group_id — two-column grouping.
  2. CPC = SUM(cost) / SUM(clicks) — aggregate both before dividing.
  3. Wrap in NULLIF(SUM(clicks), 0) to avoid division-by-zero errors.
  4. ROUND(..., 2) for 2 decimal places.
SQL
Solution
SELECT   campaign_id,
         ad_group_id,
         ROUND(
             SUM(cost) / NULLIF(SUM(clicks), 0),
         2) AS avg_cpc
FROM     ad_clicks
GROUP BY campaign_id, ad_group_id
ORDER BY campaign_id, ad_group_id;

Why SUM(cost) / SUM(clicks), not AVG(cost/clicks)?

Dividing aggregates — SUM(cost)/SUM(clicks) — gives the true weighted average CPC across the group. AVG(cost/clicks) would average the per-row ratios, which is mathematically different and usually wrong.

Medium Ad Click-Through & Conversion Rates

For each ad, calculate total clicks, total conversions, click-through rate (clicks ÷ impressions × 100), and conversion rate (conversions ÷ clicks × 100). Round rates to 2 decimal places.

ad_performance
columntype
ad_idinteger
impressionsinteger
clicksinteger
conversionsinteger
Approach
  1. SELECT ad_id, clicks, and conversions directly.
  2. CTR = clicks * 100.0 / NULLIF(impressions, 0) — multiply by 100.0 (not 100) to force decimal division.
  3. Conversion rate = conversions * 100.0 / NULLIF(clicks, 0).
  4. Wrap both in ROUND(..., 2).
SQL
Solution
SELECT
    ad_id,
    clicks,
    conversions,
    ROUND(clicks * 100.0 / NULLIF(impressions, 0), 2) AS ctr,
    ROUND(conversions * 100.0 / NULLIF(clicks, 0),      2) AS conversion_rate
FROM  ad_performance;

Integer division trap

clicks / impressions in SQL returns 0 when both are integers and clicks < impressions. Always multiply by 100.0 (float) first, or cast: clicks::numeric / impressions.

For each customer, return their most recent purchase details — customer_id, first_name, last_name, product, and purchase_date.

purchase_datedate
Approach — "Top N per group" pattern (Day 6)
  1. Open a CTE. Inside it, use ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY purchase_date DESC) to rank each customer's purchases newest-first.
  2. In the outer query, filter WHERE rn = 1 to keep only the most recent purchase per customer.

WITH latest AS (
    SELECT
        customer_id,
        product,
        purchase_date,
        ROW_NUMBER() OVER (
            PARTITION BY customer_id
            ORDER BY     purchase_date DESC
        ) AS rn
    FROM  playstore_purchases
)

    c.customer_id,
    c.first_name,
    c.last_name,
    l.product,

FROM     customers c
JOIN     latest    l ON c.customer_id = l.customer_id
WHERE    l.rn = 1;
Medium Odd & Even Measurements

The capstone problem from Day 7. Within each day, measurements arrive in time order. Sum the values at odd positions (1st, 3rd, 5th…) and even positions (2nd, 4th, 6th…) separately. Output one row per day with measurement_day, odd_sum, even_sum.

measurements
columntype
measurement_idinteger
measurement_valuedecimal
measurement_timetimestamp
Approach — three tools combined
  1. CTE: use ROW_NUMBER() OVER (PARTITION BY DATE(measurement_time) ORDER BY measurement_time) to number each measurement within its day.
  2. Outer query: GROUP BY measurement_day.
  3. CASE WHEN + modulo: SUM(CASE WHEN position % 2 = 1 THEN value ELSE 0 END) for odd_sum, same with = 0 for even_sum.
SQL
Solution
WITH numbered AS (
    SELECT
        measurement_value,
        DATE(measurement_time) AS measurement_day,
        ROW_NUMBER() OVER (
            PARTITION BY DATE(measurement_time)
            ORDER BY     measurement_time
        ) AS position
    FROM measurements
)
SELECT
    measurement_day,
    SUM(CASE WHEN position % 2 = 1
             THEN measurement_value ELSE 0
        END) AS odd_sum,
    SUM(CASE WHEN position % 2 = 0
             THEN measurement_value ELSE 0
        END) AS even_sum
FROM     numbered
GROUP BY measurement_day
ORDER BY measurement_day;
Hard Median Google Search Frequency

Calculate the median number of searches per user, rounded to 1 decimal place. The tricky part: the table is pre-aggregated — each row represents many users with the same search count.

search_frequency
columntype / meaning
searchesinteger — number of searches per session
num_usersinteger — how many users had this many searches
searchesnum_usersmeaning
122 users searched exactly once
233 users searched exactly twice
311 user searched 3 times
Approach
  1. The data is aggregated — you need to "expand" it. Row (searches=2, num_users=3) represents 3 individual data points each equal to 2.
  2. Use CROSS JOIN GENERATE_SERIES(1, num_users) to repeat each row the correct number of times, creating one row per user.
  3. Then apply PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY searches) — this is the SQL median function.
  4. Cast to numeric and ROUND(..., 1).

New functions here

GENERATE_SERIES(1, n) generates n rows (PostgreSQL). PERCENTILE_CONT(0.5) returns the exact median interpolating between values. This is a Hard problem because it requires knowing these less-common functions.

SQL
Solution
WITH expanded AS (
    SELECT   searches
    FROM     search_frequency
    CROSS JOIN GENERATE_SERIES(1, num_users)  -- one row per user
)
SELECT
    ROUND(
        PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY searches)::numeric,
    1) AS median
FROM  expanded;

If GENERATE_SERIES isn't available

In MySQL or older systems, you can use a numbers table or write a manual median with ROW_NUMBER() — count total rows, find the middle position(s), then average them. That approach is more complex but demonstrates the same logical thinking.
