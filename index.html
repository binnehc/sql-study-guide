Easy Amazon High-Value Completed Orders

Return order_id, customer_id, and total_amount for all orders that are completed, have a total above $500, and were placed in 2024. Order by amount descending.

orders
columntype
order_idinteger
customer_idinteger
order_datedate
total_amountdecimal
statusvarchar — 'completed', 'pending', 'cancelled'
Approach
  1. SELECT the three columns asked for — don't SELECT *.
  2. Filter status = 'completed' in WHERE.
  3. Add AND total_amount > 500.
  4. Add AND EXTRACT(YEAR FROM order_date) = 2024 for the year filter.
  5. ORDER BY total_amount DESC.
SQL
Solution
SELECT  order_id, customer_id, total_amount
FROM    orders
WHERE   status = 'completed'
  AND   total_amount > 500
  AND   EXTRACT(YEAR FROM order_date) = 2024
ORDER BY total_amount DESC;

EXTRACT vs date range

You could also write order_date >= '2024-01-01' AND order_date < '2025-01-01'. On indexed columns this is faster because it avoids wrapping the column in a function — both are correct in an interview.

Easy Spotify Streams per Genre

Count the total number of streams per genre. Return genre and total_streams, ordered by most streamed first.

streams
columntype
stream_idinteger
user_idinteger
song_idinteger
streamed_attimestamp
songs
columntype
song_idinteger
titlevarchar
genrevarchar
Approach
  1. JOIN streams to songs on song_id to get the genre for each stream.
  2. GROUP BY genre.
  3. COUNT(*) gives the number of streams per group.
  4. ORDER BY total_streams DESC.
SQL
Solution
SELECT   s.genre,
         COUNT(*) AS total_streams
FROM     streams  st
JOIN     songs    s  ON st.song_id = s.song_id
GROUP BY s.genre
ORDER BY total_streams DESC;
Medium Uber Riders with No Trips

Find riders who signed up more than 60 days ago but have never completed a trip. Return rider_id and name.

riders
columntype
rider_idinteger
namevarchar
signup_datedate
trips
columntype
trip_idinteger
rider_idinteger
completed_attimestamp
Approach — "missing data" pattern (Day 2)
  1. LEFT JOIN riders to trips on rider_id — every rider is kept even if they have no trips.
  2. WHERE trips.rider_id IS NULL — keeps only riders with no matching trip row.
  3. Add AND r.signup_date <= CURRENT_DATE - INTERVAL '60 days' to filter by account age.
SQL
Solution
SELECT  r.rider_id, r.name
FROM    riders r
LEFT JOIN trips t ON r.rider_id = t.rider_id
WHERE   t.rider_id IS NULL
  AND   r.signup_date <= CURRENT_DATE - INTERVAL '60 days';

Why LEFT JOIN + IS NULL?

INNER JOIN would drop riders with no trips entirely. LEFT JOIN keeps them — then IS NULL on the right table's column filters to only those with no match. This "missing data" pattern comes up constantly in analytics.

Medium Shopify High-Revenue Product Categories

Find product categories where total revenue exceeds $50,000. Return category and total_revenue rounded to 2 decimal places, sorted by revenue descending.

order_items
columntype
item_idinteger
order_idinteger
product_idinteger
quantityinteger
unit_pricedecimal
products
columntype
product_idinteger
namevarchar
categoryvarchar
Approach
  1. JOIN order_items to products on product_id.
  2. Revenue per row = quantity * unit_price — compute inside SUM.
  3. GROUP BY category, then SUM(quantity * unit_price).
  4. HAVING SUM(...) > 50000 — filter groups after aggregation.
  5. Wrap in ROUND(..., 2) and ORDER BY DESC.
SQL
Solution
SELECT   p.category,
         ROUND(SUM(oi.quantity * oi.unit_price), 2) AS total_revenue
FROM     order_items oi
JOIN     products    p  ON oi.product_id = p.product_id
GROUP BY p.category
HAVING   SUM(oi.quantity * oi.unit_price) > 50000
ORDER BY total_revenue DESC;

Can't use the alias in HAVING

HAVING runs before SELECT, so the alias total_revenue doesn't exist yet. You must repeat the full expression — or wrap the whole thing in a CTE and filter in the outer query.

Medium LinkedIn Top Earner per Department

Return the highest-paid employee in each department. If two employees tie, include both. Return employee_id, full_name, department, and salary.

employees
columntype
employee_idinteger
full_namevarchar
departmentvarchar
salarydecimal
Approach — "top N per group" (Day 6)
  1. In a CTE, use DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) to rank employees within each department.
  2. Use DENSE_RANK (not ROW_NUMBER) so tied salaries both get rank 1.
  3. Outer query: WHERE dr = 1 to keep only top earners.
SQL
Solution
WITH ranked AS (
    SELECT
        employee_id, full_name, department, salary,
        DENSE_RANK() OVER (
            PARTITION BY department
            ORDER BY     salary DESC
        ) AS dr
    FROM employees
)
SELECT  employee_id, full_name, department, salary
FROM    ranked
WHERE   dr = 1
ORDER BY department;

DENSE_RANK vs ROW_NUMBER for ties

ROW_NUMBER gives every row a unique number — two tied employees get ranks 1 and 2, so only one would be kept by WHERE rn = 1. DENSE_RANK gives them both rank 1, so both are returned. Pick the one that matches what the question asks.

Medium DoorDash Weekend vs Weekday Revenue

For each restaurant, calculate total weekday revenue and total weekend revenue in a single row per restaurant. Saturday and Sunday count as weekend.

orders
columntype
order_idinteger
restaurant_idinteger
order_timetimestamp
total_amountdecimal
Approach — pivot with CASE WHEN
  1. Use EXTRACT(DOW FROM order_time) to get the day of week (0 = Sunday, 6 = Saturday).
  2. Inside SUM(), use CASE WHEN DOW IN (0, 6) THEN total_amount ELSE 0 END for weekend revenue.
  3. Repeat with NOT IN (0, 6) for weekday revenue.
  4. GROUP BY restaurant_id.
SQL
Solution
SELECT
    restaurant_id,
    ROUND(SUM(CASE WHEN EXTRACT(DOW FROM order_time) IN (0, 6)
                  THEN total_amount ELSE 0 END), 2) AS weekend_revenue,
    ROUND(SUM(CASE WHEN EXTRACT(DOW FROM order_time) NOT IN (0, 6)
                  THEN total_amount ELSE 0 END), 2) AS weekday_revenue
FROM   orders
GROUP BY restaurant_id
ORDER BY restaurant_id;

Pivoting with CASE WHEN inside SUM

Instead of getting two rows per restaurant (one for each day type), CASE WHEN inside SUM produces two columns in one row — a very common interview pattern for conditional aggregation.

Medium Google Odd & Even Measurements

The capstone problem from Day 7. Within each day, measurements arrive in time order. Sum the values at odd positions (1st, 3rd, 5th…) and even positions (2nd, 4th, 6th…) separately. Output one row per day with measurement_day, odd_sum, even_sum.

measurements
columntype
measurement_idinteger
measurement_valuedecimal
measurement_timetimestamp
Approach — three tools combined
  1. CTE: use ROW_NUMBER() OVER (PARTITION BY DATE(measurement_time) ORDER BY measurement_time) to number each measurement within its day.
  2. Outer query: GROUP BY measurement_day.
  3. CASE WHEN + modulo: SUM(CASE WHEN position % 2 = 1 THEN value ELSE 0 END) for odd_sum, same with = 0 for even_sum.
SQL
Solution
WITH numbered AS (
    SELECT
        measurement_value,
        DATE(measurement_time) AS measurement_day,
        ROW_NUMBER() OVER (
            PARTITION BY DATE(measurement_time)
            ORDER BY     measurement_time
        ) AS position
    FROM measurements
)
SELECT
    measurement_day,
    SUM(CASE WHEN position % 2 = 1
             THEN measurement_value ELSE 0
        END) AS odd_sum,
    SUM(CASE WHEN position % 2 = 0
             THEN measurement_value ELSE 0
        END) AS even_sum
FROM     numbered
GROUP BY measurement_day
ORDER BY measurement_day;
Hard Airbnb Cumulative Booking Revenue

For each month in 2024, show monthly_revenue and running_total — the cumulative revenue from January through that month. Round both to 2 decimal places.

bookings
columntype
booking_idinteger
host_idinteger
check_in_datedate
total_pricedecimal
Approach — running total with a window function
  1. CTE: GROUP BY month using DATE_TRUNC('month', check_in_date), SUM total_price to get monthly revenue.
  2. Outer query: use SUM(monthly_revenue) OVER (ORDER BY month ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) to accumulate.
  3. The window function runs across rows without collapsing them — each row keeps its monthly figure AND gets a running total column.

Why not just GROUP BY again?

A running total requires looking at previous rows. Regular GROUP BY collapses rows — you'd lose the per-month breakdown. Window functions keep all rows and let you "look back" using ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW.

SQL
Solution
WITH monthly AS (
    SELECT
        DATE_TRUNC('month', check_in_date) AS month,
        ROUND(SUM(total_price), 2)             AS monthly_revenue
    FROM    bookings
    WHERE   EXTRACT(YEAR FROM check_in_date) = 2024
    GROUP BY DATE_TRUNC('month', check_in_date)
)
SELECT
    month,
    monthly_revenue,
    ROUND(SUM(monthly_revenue) OVER (
        ORDER BY month
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ), 2) AS running_total
FROM   monthly
ORDER BY month;

ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW

This window frame means "from the first row in the partition up to and including the current row." For SUM() OVER(ORDER BY ...) this is actually the default — but writing it explicitly shows the interviewer you know what's happening.
