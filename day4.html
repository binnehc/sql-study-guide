Part 1 · NULL Handling

NULL means "unknown" or "missing" — it is not zero, not an empty string. NULL has its own rules that trip up almost every SQL beginner.

NULL ≠ NULL

You cannot compare NULL with =. WHERE col = NULL always returns nothing. Always use IS NULL or IS NOT NULL.

COUNT(*) vs COUNT(col)

COUNT(*) counts all rows. COUNT(col) counts only non-NULL values in that column. These behave differently whenever NULLs are present — a very common source of subtle bugs.

NULLs in JOINs

NULL never matches anything — not even another NULL. A row with a NULL key will never match in a JOIN condition.

COALESCE()

COALESCE(a, b, c, ...) returns the first non-NULL value from its arguments, left to right. COALESCE(col, 0) returns col if it's not NULL, otherwise 0.

SQL · NULL Gotchas
-- WRONG — this returns 0 rows, not the NULLs
SELECT * FROM employees WHERE dept_id = NULL;

-- CORRECT
SELECT * FROM employees WHERE dept_id IS NULL;

-- COUNT difference: employees table has 4 rows, 1 has NULL dept_id
SELECT COUNT(*)         -- returns 4  (all rows)
     , COUNT(dept_id)   -- returns 3  (ignores NULLs)
FROM employees;

-- Replace NULL with a default
SELECT name, COALESCE(dept_id, 0) AS dept_id
FROM   employees;
Part 2 · Subqueries

A subquery is a query nested inside another query. They work but can get hard to read — CTEs (Day 5) are the cleaner alternative. Knowing subqueries helps you read legacy code and understand how CTEs improve on them.

① Inner query — runs first
SELECT AVG(salary) FROM employees
evaluates to a single value → 97,000
result passed to outer query ↓
② Outer query — uses the result
WHERE salary > 97,000
returns only employees above the average
SQL · Subqueries
-- In WHERE: employees earning above the average
SELECT name, salary
FROM   employees
WHERE  salary > (SELECT AVG(salary) FROM employees);

-- In FROM: treat result as a temporary table (like a CTE)
SELECT dept, avg_sal
FROM (
    SELECT department AS dept, AVG(salary) AS avg_sal
    FROM   employees
    GROUP BY department
) AS dept_avgs
WHERE  avg_sal > 90000;
Part 3 · Self-Joins

A self-join joins a table to itself. This sounds strange but is very useful for comparing rows within the same table — classic examples: finding employees who earn more than their manager, or finding pairs of products bought together.

SQL · Self-Join: employees earning more than their manager
-- Alias the same table twice — "e" is the employee, "m" is their manager
SELECT
    e.name        AS employee,
    e.salary      AS emp_salary,
    m.name        AS manager,
    m.salary      AS mgr_salary
FROM       employees e
INNER JOIN employees m ON e.manager_id = m.employee_id
WHERE      e.salary > m.salary;

Self-Join Pattern

Always give the table two different aliases (e.g., e and m). SQL then treats them as if they were two separate tables. The join condition defines the relationship between the two copies.

Day 4 mastery checklist

  • I always use IS NULL / IS NOT NULL (never = NULL)
  • I can write a self-join by aliasing a table twice
  • I can use COALESCE to replace NULL with a fallback value

Practice

Write a query to find employees whose salary is higher than their manager's salary. Return the employee name, their salary, and their manager's salary.

Table: employees (employee_id, name, salary, manager_id) — manager_id references employee_id

SQL
Answer
SELECT
    e.name   AS employee,
    e.salary AS emp_salary,
    m.salary AS mgr_salary
FROM       employees e
INNER JOIN employees m ON e.manager_id = m.employee_id
WHERE      e.salary > m.salary;
